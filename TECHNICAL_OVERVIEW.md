# TECHNICAL OVERVIEW: Cold Storage AI Monitoring System

This document details the architecture, data flow, and CI/CD pipeline for the Cold Storage AI Monitor project.

---

## 1. System Architecture

The system is designed with a **Decoupled Edge-to-Cloud** architecture, ensuring that heavy computation (AI inference) and data storage are handled by robust cloud infrastructure while the edge device focuses on real-time data collection.

### Core Components
| Component | Technology | Primary Role |
|-----------|------------|--------------|
| **Edge Hardware** | ESP32 Microcontroller | Sensor data acquisition & data packetization. |
| **Sensors** | DHT11, MQ2, MQ135 | Environment monitoring (Temp, Hum, Smoke, Gas). |
| **Backend API** | FastAPI (Python) | Data ingestion, Security, & AI Model Hosting. |
| **ML Engine** | Logistic Regression | Real-time environment safety classification. |
| **Database** | SQLite (SQLAlchemy) | Persistent historical reading storage. |
| **Real-time Comms** | WebSockets | Instant dashboard updates without polling. |
| **Frontend** | Vanilla JS + Chart.js | Visualization and user alerting interface. |
| **Deployment** | Docker + Railway | Containerized cloud hosting and CI/CD. |

---

## 2. End-to-End Data Flow

The data follows a unidirectional path from physical sensors to the user's dashboard:

1.  **Sensing**: DHT11 and MQ sensors generate analog/digital signals relative to the cold storage environment.
2.  **Processing (Edge)**: The ESP32 reads these pins every 2 seconds, converts them to standard values (Celsius, Humidity %, PPM), and creates a JSON payload.
3.  **Transmission**: The ESP32 sends an **HTTP POST** request with an `x-api-key` header to the Railway production endpoint (`/api/ingest`).
4.  **Inference (Cloud)**:
    *   FastAPI receives the JSON.
    *   The **Predictor Module** applies a pre-trained **Logistic Regression** model logic:
        *   `Normalize Inputs` -> `Dot Product (Weights)` -> `Add Bias` -> `Sigmoid Activation`.
    *   The result (SAFE/DANGER) is determined.
5.  **Persistence**: The reading and the AI prediction are saved into the `cold_storage.db` SQLite database.
6.  **Broadcasting**: The update is pushed through an active **WebSocket** channel to every connected browser.
7.  **Visualization**: The Dashboard (HTML/JS) receives the WebSocket packet and updates the charts instantly.

## 3. CI/CD Pipeline (DevOps)

The project implements a "Zero-Touch" deployment strategy using GitHub and Railway.

### The Workflow:
1.  **Local Development**: Code is written and tested on the local machine. ML model weights are generated by `pipeline.py`.
2.  **Version Control (Git)**: Code is committed and pushed to the **Main Branch** on GitHub.
3.  **Webhook Trigger**: GitHub notifies Railway of the new commit.
4.  **Automated Build**:
    *   Railway detects the `Dockerfile` in the root.
    *   It initiates a multi-stage **Docker Build**.
    *   Dependencies are installed via `pip` from `requirements.txt`.
    *   Static assets (frontend) and ML outputs (results.json) are packaged into the image.
5.  **Provisioning**: Railway launches the container, binds it to the provided `$PORT`, and assigns a secure SSL/TLS public URL.
6.  **Live Updates**: The service becomes instantly available to the ESP32 and the Web users.

---

## 4. Key Security Features
*   **Static Internal API Key**: Prevents rogue devices from flooding the database with fake sensor readings.
*   **Decoupled Webhooks**: The ESP32 only needs to know the target URL; it does not require access to the database or internal server logic.
*   **CORS Protection**: The API is restricted to allow only authenticated interactions.
